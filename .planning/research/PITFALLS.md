# Domain Pitfalls

**Domain:** Book summaries PWA with payments, subscriptions, freemium access, and offline content delivery (Russian market)
**Researched:** 2026-03-18

## Critical Pitfalls

Mistakes that cause security breaches, revenue loss, or require rewrites.

### Pitfall 1: Users Can Self-Promote to Pro via Supabase RLS

**What goes wrong:** The current `profiles` table RLS policy allows any authenticated user to UPDATE their own row without column restrictions. A user can open browser DevTools and run `supabase.from('profiles').update({ subscription_type: 'pro_monthly', subscription_expires_at: '2099-01-01' }).eq('user_id', userId)` to get free Pro access indefinitely.

**Why it happens:** The migration `20260314171856` creates a blanket UPDATE policy: `USING (auth.uid() = user_id)` with no column restrictions. This is fine for `name` and `avatar_url`, but catastrophic for `subscription_type` and `subscription_expires_at`.

**Consequences:** Complete revenue bypass. Any technically inclined user (and your target audience of entrepreneurs/managers includes many) can get free Pro. Once shared on forums, all users do it.

**Prevention:**
- Split the UPDATE policy into column-specific policies: users can update `name`, `avatar_url` only
- Subscription fields must ONLY be writable by a service role (Edge Function) that processes payment webhooks
- Create a separate `subscriptions` table managed exclusively by server-side code with no user-facing UPDATE policy
- Add a database trigger or check constraint that prevents `subscription_type` changes unless called from a SECURITY DEFINER function

**Detection:** Audit RLS policies before any payment integration. Check if any user-updatable table contains payment/subscription fields. Test by attempting to update subscription fields from the client.

**Phase:** Must be fixed BEFORE payment integration begins. This is a prerequisite, not a cleanup task.

### Pitfall 2: Client-Side-Only Access Control

**What goes wrong:** The `useAccessControl` hook enforces freemium limits (5 free reads, audio for Pro only, downloads for Pro only) entirely on the client side. The actual book content, audio URLs, and full text are still served from Supabase to any authenticated user -- the client just chooses not to display them.

**Why it happens:** The codebase has no server-side content gating. All book content is fetched via `supabase.from('books')` and `supabase.from('book_summaries')` with RLS policies that only check authentication, not subscription status. Audio files in Supabase Storage are accessible to any authenticated user who knows the URL.

**Consequences:** Users bypass the paywall by: (a) directly querying the Supabase API, (b) inspecting network requests to find audio URLs, (c) disabling the React hook's logic via browser extensions. All premium content is effectively free.

**Prevention:**
- Implement server-side access control via Supabase RLS policies that check `subscription_type` on the `profiles` table before returning premium content
- For audio files: use signed URLs generated by an Edge Function that verifies subscription status, instead of direct Storage URLs
- For text content: gate full summary content behind an RLS policy or Edge Function that checks subscription
- Allow free-tier users to see book metadata (title, cover, description) but not full content
- Keep client-side checks for UX (showing paywall prompts) but never rely on them for enforcement

**Detection:** Try accessing premium content by directly calling the Supabase client from DevTools while on a free account. If content is returned, access control is broken.

**Phase:** Must be implemented during payment/freemium phase. Server-side gating is the foundation -- client-side UX is just polish on top.

### Pitfall 3: YooKassa Webhook Security and Idempotency

**What goes wrong:** YooKassa sends payment confirmation via webhooks to your server endpoint. Common mistakes: (a) not verifying webhook signatures, allowing forged payment confirmations; (b) not handling duplicate webhooks, causing double subscription extensions; (c) processing webhooks synchronously and timing out, causing YooKassa to retry and create duplicate processing.

**Why it happens:** YooKassa's webhook documentation is sparse compared to Stripe. Developers often skip signature verification because "it works in testing." YooKassa retries failed webhooks multiple times, and without idempotency keys, each retry can trigger a new subscription period.

**Consequences:** (a) Forged webhooks = free Pro access. (b) Duplicate processing = accounting errors, extended subscriptions beyond what was paid. (c) Timeout retries = race conditions between webhook handlers.

**Prevention:**
- Always verify the webhook IP address (YooKassa sends from specific IP ranges) and/or use the notification object's `event` field
- Store a `payment_id` from YooKassa in a `payments` table and check for duplicates before processing
- Use database transactions: check if payment_id exists, if not insert it and update subscription, all in one transaction
- Return HTTP 200 immediately, process asynchronously if needed
- Implement a `payments` table with columns: `id`, `yookassa_payment_id` (UNIQUE), `user_id`, `amount`, `status`, `created_at`
- Log every webhook receipt for audit purposes

**Detection:** Send the same webhook payload twice in testing. If subscription extends twice, idempotency is broken. Check if unauthenticated POST to webhook endpoint can modify subscriptions.

**Phase:** Payment integration phase. Must be correct from day one -- payment bugs erode user trust and are hard to fix retroactively.

### Pitfall 4: Offline Content Persists After Subscription Expires

**What goes wrong:** A Pro user downloads 50 book summaries and audio files to IndexedDB. Their subscription expires. The downloaded content remains fully accessible offline forever because the offline reader (`OfflineReaderPage`) reads from IndexedDB without checking subscription status.

**Why it happens:** Offline-first architecture intentionally avoids network checks. The download was legitimate at the time. No mechanism exists to revoke or expire cached content.

**Consequences:** Users subscribe for one month, download everything, cancel. Effective piracy of your entire catalog. Revenue model collapses because the "correct" behavior (subscribe once, download all, cancel) is the optimal user strategy.

**Prevention:**
- Store a `subscriptionValidUntil` timestamp alongside downloaded content metadata
- On app launch (when online), sync subscription status and update the validity window
- When offline, check the stored validity timestamp -- allow access only within a grace period (e.g., 7 days past expiry)
- After grace period, show a "renew subscription to access offline content" prompt instead of content
- Do NOT delete content on expiry (users who re-subscribe expect their downloads to still be there) -- just gate access
- Consider a "lease" model: downloads are valid for N days and must be refreshed periodically while subscribed

**Detection:** Subscribe, download content, cancel subscription, go offline, check if content is still accessible. If yes, this pitfall is present.

**Phase:** Must be designed during payment/offline integration. Retrofitting content expiry into existing offline storage is much harder than building it in from the start.

### Pitfall 5: No Payment State Machine -- Subscription Status Becomes Inconsistent

**What goes wrong:** Subscription lifecycle has many states: pending payment, active, past due, cancelled but still active until period end, expired, refunded. Treating it as a simple boolean (`isPro = true/false`) leads to impossible states: user paid but shows as free (webhook delay), user cancelled but still has access until period end (not handled), user's payment failed on renewal but app still shows Pro.

**Why it happens:** The current `useSubscription` hook checks `subscription_type !== 'free' && !isExpired` which is a simplified binary check. Real payment systems have at least 5-6 distinct states. YooKassa's auto-payment (recurring) adds complexity with `waiting_for_capture`, `succeeded`, `canceled` states.

**Consequences:** Users see incorrect subscription status. Support tickets flood in. Worst case: users who paid can't access content, or users who didn't pay can access it.

**Prevention:**
- Create a proper subscription state machine with states: `free`, `pending`, `active`, `past_due`, `cancelled_active` (cancelled but paid through period end), `expired`
- Store `subscription_status` separately from `subscription_type` (plan type vs current status)
- Add `subscription_started_at`, `subscription_cancelled_at`, `next_billing_date` columns
- Create a dedicated `subscriptions` table rather than storing on `profiles`
- Handle YooKassa's specific payment statuses: map each to your internal state
- Add a daily cron job (Supabase pg_cron or Edge Function) to expire subscriptions whose `subscription_expires_at` has passed

**Detection:** Walk through every subscription lifecycle scenario on paper: new subscription, renewal, failed renewal, cancellation mid-period, refund, re-subscription after expiry. If any scenario produces an ambiguous state, the model is incomplete.

**Phase:** Payment integration phase. The data model must be designed before the first payment is processed.

## Moderate Pitfalls

### Pitfall 6: Audio Download Fails Silently on Large Files

**What goes wrong:** The current `saveAudioOffline` function streams audio via `fetch()` and stores it as a Blob in IndexedDB. On mobile browsers with limited memory, downloading a 50-100MB audio file can cause an out-of-memory crash or a silent failure where the blob is truncated.

**Prevention:**
- Implement chunked storage: split large audio files into chunks (e.g., 5MB each) and store individually in IndexedDB
- Add integrity verification: store expected file size in metadata, verify stored blob size matches after download
- Add download resumption: if a download fails mid-way, resume from the last successfully stored chunk
- Set a maximum single-file size (e.g., 50MB) and warn users before downloading larger files
- Consider compressing audio to lower bitrates for offline (64kbps mono is sufficient for spoken word)

**Phase:** Offline enhancement phase. The current implementation works for small files but will break as the catalog grows to include longer books with larger audio files.

### Pitfall 7: ObjectURL Memory Leaks in Offline Audio Playback

**What goes wrong:** The `getAudioOffline` function creates a new `URL.createObjectURL(blob)` every time it's called but never revokes old URLs. Each call allocates memory for the blob that is never freed until the page is unloaded.

**Prevention:**
- Track created ObjectURLs and call `URL.revokeObjectURL()` when the audio source changes or component unmounts
- Consider using a single ObjectURL per audio session, revoking on book switch
- Add cleanup logic in `AudioContext.tsx` when switching between online and offline audio sources

**Phase:** Offline enhancement phase. Memory leaks compound over time and cause app crashes on memory-constrained mobile devices.

### Pitfall 8: YooKassa Recurring Payments Require Separate Flow

**What goes wrong:** Developers assume YooKassa's subscription is like Stripe's -- create a subscription object and it auto-renews. YooKassa does not have a "subscription" primitive. You must implement recurring payments yourself using saved payment methods and scheduled charges.

**Prevention:**
- Use YooKassa's "save payment method" flag (`save_payment_method: true`) on initial payment to save the user's card
- Store the `payment_method_id` returned after successful first payment
- Implement a server-side cron job (Supabase pg_cron or external scheduler) that creates new payments using the saved method before each billing period
- Handle `payment_method` expiry and card updates
- Implement a grace period for failed recurring charges (retry 3 times over 3 days before downgrading)
- Build a manual "renew subscription" fallback for when auto-payment fails

**Phase:** Payment integration phase. This is an architectural decision that affects the entire payment flow -- cannot be added later without rewriting.

### Pitfall 9: Storage Quota Mismatch Between Estimated and Actual Size

**What goes wrong:** The `useDownloads` hook estimates download size using `audioSizeHint` (from `content-length` header) before downloading. But: (a) `content-length` may not be present for chunked responses from Supabase Storage, (b) the actual stored blob size in IndexedDB includes overhead, (c) the quota check uses metadata-tracked sizes which can drift from actual IndexedDB usage.

**Consequences:** User passes the quota check, starts downloading, and the download fails mid-way when IndexedDB actually runs out of space. Partial data is stored but metadata says download succeeded.

**Prevention:**
- Use `navigator.storage.estimate()` to get actual browser-reported quota and usage
- Add post-download verification: compare actual stored size with metadata
- Implement atomic downloads: if any part fails, clean up all partial data
- Add a safety margin (e.g., require 20% free space beyond the estimated download size)
- Handle `QuotaExceededError` specifically in the catch block of storage operations

**Phase:** Offline enhancement phase.

### Pitfall 10: Free Read Counter is Gameable

**What goes wrong:** The `useAccessControl` hook counts free reads by querying `user_progress` entries. A user can: (a) create a new account to get 5 more free reads, (b) the count is based on `user_progress` rows, so if a user reads without creating progress (possible if they read fast and close before progress saves), the read doesn't count.

**Prevention:**
- Track free reads server-side in a dedicated counter on the `profiles` table, incremented by a database trigger when a new `user_progress` entry is created
- Use device fingerprinting (localStorage token) to limit free reads across accounts on the same device
- Consider making the free tier time-limited (e.g., "first 7 days free") rather than count-limited, which is harder to game
- Increment the counter on first open of a book, not on progress save, to prevent the "read without saving" bypass

**Phase:** Freemium implementation phase.

## Minor Pitfalls

### Pitfall 11: Service Worker Cache Serves Stale Paywall State

**What goes wrong:** After a user subscribes, the Service Worker may serve cached HTML/JS that still contains the old `useSubscription` state (free). User sees paywall prompts even though they just paid. Requires force-refresh or clearing cache.

**Prevention:**
- Ensure subscription status is always fetched from the network (React Query already does this, but verify `staleTime` is short)
- Add a `skipWaiting` + `clients.claim` strategy for Service Worker updates when critical state changes
- After successful payment, force invalidate the `subscription` query key

**Phase:** Payment integration phase.

---

### Pitfall 12: Price Display Hardcoded in Multiple Places

**What goes wrong:** The `PaywallPrompt.tsx` component has hardcoded prices ("299 p/mes", "1 990 p/god"). When prices change (for promotions, A/B testing, or inflation), developers must find and update every hardcoded instance. Missing one creates conflicting price displays.

**Prevention:**
- Store pricing in a Supabase table or Edge Function config
- Create a single `PRICING` constant object imported by all components
- At minimum, centralize all price strings in one file

**Phase:** Payment integration phase.

---

### Pitfall 13: Offline-Online Sync Conflict for Highlights and Progress

**What goes wrong:** User creates highlights and reading progress while offline. Meanwhile, on another device (or after re-login), they also make changes. When the offline device reconnects, the sync logic must merge conflicting states. The current `useOfflineSync` has no conflict resolution strategy.

**Prevention:**
- Implement "last write wins" with timestamps as a minimum viable strategy
- For highlights: use unique IDs generated client-side (UUIDs) to avoid duplicate creation
- For reading progress: always take the higher percentage/further position (user never wants to go backward)
- Queue offline mutations with timestamps and replay them on reconnection in order

**Phase:** Offline enhancement phase.

---

### Pitfall 14: Supabase Storage URLs Expose Direct Access to Audio Files

**What goes wrong:** Supabase Storage public URLs are predictable and don't expire. If audio file URLs are logged, shared, or inspected via DevTools, they can be shared freely and accessed without authentication. Premium audio content becomes freely distributable.

**Prevention:**
- Use private buckets with signed URLs that expire (e.g., 1 hour validity)
- Generate signed URLs via an Edge Function that verifies subscription status
- Never expose raw Storage URLs in client-side API responses -- always proxy through a signing function
- Set bucket policies to prevent public access

**Phase:** Payment integration phase. Must be in place before premium content exists.
