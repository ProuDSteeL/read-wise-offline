# Phase 1: Security Hardening - Research

**Researched:** 2026-03-18
**Domain:** Supabase RLS, Edge Functions, Storage signed URLs, server-side content gating
**Confidence:** HIGH

## Summary

Phase 1 addresses three critical security vulnerabilities in the ReadWise Offline PWA: (1) the `profiles` table has a blanket UPDATE RLS policy that allows any authenticated user to self-promote to Pro by modifying `subscription_type` and `subscription_expires_at` via DevTools, (2) all content gating (free read limits, audio access) is client-side only -- the Supabase API returns full content to any authenticated user regardless of subscription status, and (3) audio files are stored in a public bucket with predictable, non-expiring URLs that can be freely shared.

The solution involves a database trigger to protect subscription fields on `profiles`, a new Edge Function to serve summary content with server-side truncation for non-Pro users, and converting the audio bucket to private with signed URLs generated by another Edge Function. The existing `useAccessControl` hook continues to drive UI decisions but the server becomes the enforcement layer.

**Primary recommendation:** Implement all three security layers (trigger + content gating Edge Function + audio signing Edge Function) before any payment integration, as they form the trust foundation for monetization.

### Pattern 1: Database Trigger for Field Protection (SEC-01)
**What:** A BEFORE UPDATE trigger on `profiles` that silently resets protected fields to their OLD values when the caller is not `service_role`.
**When to use:** When you need to allow UPDATE on a row but restrict specific columns.
**Why trigger over RLS:** Supabase RLS operates at the row level. You cannot have an RLS policy that says "allow UPDATE on columns A,B but deny UPDATE on columns C,D." The trigger approach lets the UPDATE succeed (so name/avatar updates work) while silently preserving subscription fields.

**Key implementation detail:** In Supabase, the service_role key sets the PostgreSQL role to `service_role` at the session level. Edge Functions using `SUPABASE_SERVICE_ROLE_KEY` bypass RLS entirely. The trigger must check if the current session is NOT service_role to apply protection. The check uses `current_setting('role')` which returns `'authenticated'` for regular users and is bypassed entirely for `service_role` (since service_role bypasses RLS and runs as `postgres` superuser -- but triggers still fire). The correct approach: check `current_user` or `session_user` -- when service_role key is used, the role is `service_role`; when anon/authenticated key is used, it is `authenticated` or `anon`.

```sql
CREATE OR REPLACE FUNCTION public.protect_subscription_fields()
RETURNS TRIGGER AS $$
BEGIN
  -- service_role bypasses this check
  -- For authenticated users, preserve subscription fields
  IF current_setting('role', true) IS DISTINCT FROM 'service_role' THEN
    NEW.subscription_type := OLD.subscription_type;
    NEW.subscription_expires_at := OLD.subscription_expires_at;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER SET search_path = public;

CREATE TRIGGER protect_subscription_fields_trigger
  BEFORE UPDATE ON public.profiles
  FOR EACH ROW EXECUTE FUNCTION public.protect_subscription_fields();
```

**IMPORTANT NOTE on role detection:** The CONTEXT.md mentions protecting a `role` field, but the `profiles` table schema does not have a `role` column. Roles are stored in the separate `user_roles` table which already has no user-facing UPDATE policy (only SELECT). The trigger should protect only `subscription_type` and `subscription_expires_at`.

### Pattern 2: Edge Function for Content Gating (SEC-02)
**What:** A `get-summary` Edge Function that acts as a proxy between the client and the `summaries` table, applying server-side truncation logic.
**When to use:** When the client requests summary content for reading.
**Flow:**
1. Client calls Edge Function with `bookId` and auth JWT
2. Edge Function extracts user from JWT via `supabase.auth.getUser()`
3. Edge Function checks subscription status from `profiles` (using service_role client)
4. Edge Function counts free reads from `user_progress`
5. If user is Pro OR has not exhausted free reads: return full content
6. Otherwise: return truncated content (first 20-30% at paragraph boundary)

### Pattern 3: Storage Signed URLs (SEC-03)
**What:** Convert `audio-files` bucket to private and serve audio through signed URLs.
**When to use:** Whenever audio playback is requested.
**Flow:**
1. Client calls `get-audio-url` Edge Function with `bookId` and auth JWT
2. Edge Function verifies user is Pro (checks `profiles.subscription_type`)
3. If Pro: generates 24-hour signed URL via `createSignedUrl(path, 86400)`
4. Returns signed URL to client
5. Client sets `audio.src` to signed URL
6. AudioContext stores the signed URL -- valid for 24h, no need to re-request during a session

### Anti-Patterns to Avoid
- **Direct Supabase query for gated content:** After this phase, `useSummary` must NOT query `summaries` table directly for content. Metadata (audio_url existence, size) can still come from the table, but the actual text content must come from the Edge Function.
- **Storing signed URLs in state permanently:** Signed URLs expire. If a user leaves the app for >24h and returns, the stored URL will fail. AudioContext should handle 403/expired errors by re-requesting a signed URL.
- **Using `getPublicUrl` for audio after bucket is private:** The admin upload flow currently uses `getPublicUrl` to store the audio URL in the `summaries.audio_url` column. After making the bucket private, `getPublicUrl` URLs will return 403. The `audio_url` column should store the file PATH (e.g., `audio/book-id.mp3`), not the full public URL.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Signed URL generation | Custom JWT/HMAC token signing | `supabase.storage.from('audio-files').createSignedUrl(path, expiresIn)` | Built-in, handles crypto, integrated with Supabase auth |
| Content truncation | Complex character counting with UTF-8 handling | Split on paragraph boundaries (double newline in Markdown) | Summaries are Markdown; splitting on `\n\n` gives clean paragraph breaks |
| JWT verification in Edge Functions | Manual JWT parsing | `supabase.auth.getUser()` with user's JWT from Authorization header | Built-in, handles token validation, expiry checking |
| Role detection in triggers | Parsing JWT claims in SQL | `current_setting('role', true)` | Supabase sets this automatically per request |

**Key insight:** Supabase provides all the primitives needed for this security layer. No external auth libraries, no custom crypto, no third-party services.

## Common Pitfalls

### Pitfall 1: audio_url Column Stores Full Public URL, Not Path
**What goes wrong:** The `summaries.audio_url` column currently stores the full public URL from `getPublicUrl()` (e.g., `https://xxx.supabase.co/storage/v1/object/public/audio-files/audio/book.mp3`). After making the bucket private, these URLs stop working. But the column is also used to check if audio EXISTS (`!!summary?.audio_url`).
**Why it happens:** `getPublicUrl` returns a full URL, and the admin form stores that directly.
**How to avoid:** Migration must extract the file path from existing `audio_url` values OR keep the full URL for existence checks but never use it for playback. The Edge Function should derive the storage path from the book ID or from a path column. Recommendation: add a migration that strips the base URL prefix from existing `audio_url` values, leaving just the path (e.g., `audio/book-id.mp3`).
**Warning signs:** Audio playback breaks for all books immediately after making the bucket private.

### Pitfall 2: Edge Function CORS Headers Missing
**What goes wrong:** The client (running on the app's domain) calls Edge Functions but gets CORS errors because the Edge Function doesn't include the proper CORS headers.
**Why it happens:** Easy to forget when creating new Edge Functions.
**How to avoid:** Copy the `corsHeaders` pattern from the existing `send-notifications` function. Handle `OPTIONS` preflight requests. The existing pattern is:
```typescript
const corsHeaders = {
  "Access-Control-Allow-Origin": "*",
  "Access-Control-Allow-Headers":
    "authorization, x-client-info, apikey, content-type, ...",
};
```

### Pitfall 3: Free Reads Counter Race Condition
**What goes wrong:** The counter in `useAccessControl` counts `user_progress` rows. But progress is saved AFTER reading starts. If the user opens 5 books simultaneously before any progress is saved, they get 5 free reads beyond the limit.
**Why it happens:** Counter is read from cached React Query data; progress is written asynchronously.
**How to avoid:** The server-side gating Edge Function is the real enforcement. It should atomically check read count AND record the read. The client counter is for UX only. The Edge Function should: (1) count existing `user_progress` rows, (2) if under limit and no existing progress for this book, INSERT a progress row, (3) return full content. This makes the read count authoritative.

### Pitfall 4: Trigger Does Not Fire for service_role
**What goes wrong:** In Supabase, when using the service_role key, queries bypass RLS but triggers still fire. If the trigger incorrectly blocks ALL updates to subscription fields (instead of only blocking non-service_role), then even Edge Functions cannot update subscriptions.
**Why it happens:** Misunderstanding of how service_role interacts with triggers vs RLS.
**How to avoid:** The trigger must check the current role and ONLY reset fields when the caller is `authenticated` (regular user). When the caller is `service_role`, let the update through. Test this by: (a) attempting to update subscription_type as a regular user (should silently revert), (b) attempting to update subscription_type via service_role (should succeed).

### Pitfall 5: Breaking Admin Audio Upload Flow
**What goes wrong:** The admin form uploads audio to the `audio-files` bucket and calls `getPublicUrl`. After making the bucket private, `getPublicUrl` still returns a URL but it won't work for unauthenticated access. The admin can still upload (has admin RLS policy) but the stored URL is useless.
**How to avoid:** Change the admin form to store just the file path in `audio_url`, not the full public URL. The Edge Function for signed URLs uses this path with `createSignedUrl`. Admin preview can use `createSignedUrl` too (admin is authenticated).

## Code Examples

Verified patterns from the existing codebase and official Supabase documentation:

### Edge Function Template (from existing send-notifications)
```typescript
// supabase/functions/get-summary/index.ts
import { createClient } from "npm:@supabase/supabase-js@2";

const corsHeaders = {
  "Access-Control-Allow-Origin": "*",
  "Access-Control-Allow-Headers":
    "authorization, x-client-info, apikey, content-type",
};

Deno.serve(async (req) => {
  if (req.method === "OPTIONS") {
    return new Response(null, { headers: corsHeaders });
  }

  try {
    // Get user from JWT
    const authHeader = req.headers.get("Authorization");
    if (!authHeader) {
      return new Response(JSON.stringify({ error: "No auth" }), {
        status: 401,
        headers: { ...corsHeaders, "Content-Type": "application/json" },
      });
    }

    const supabaseUrl = Deno.env.get("SUPABASE_URL")!;
    const serviceKey = Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!;

    // User client (to get user identity)
    const userClient = createClient(supabaseUrl, Deno.env.get("SUPABASE_ANON_KEY")!, {
      global: { headers: { Authorization: authHeader } },
    });
    const { data: { user }, error: userError } = await userClient.auth.getUser();
    if (userError || !user) {
      return new Response(JSON.stringify({ error: "Unauthorized" }), {
        status: 401,
        headers: { ...corsHeaders, "Content-Type": "application/json" },
      });
    }

    // Service client (to bypass RLS for subscription check)
    const adminClient = createClient(supabaseUrl, serviceKey);

    // ... business logic using adminClient to check subscription
    // ... return full or truncated content
  } catch (err) {
    return new Response(JSON.stringify({ error: String(err) }), {
      status: 500,
      headers: { ...corsHeaders, "Content-Type": "application/json" },
    });
  }
});
```

### Summary Truncation at Paragraph Boundary
```typescript
function truncateSummary(content: string, targetPercent: number = 0.25): string {
  const paragraphs = content.split(/\n\n+/);
  const totalLength = content.length;
  const targetLength = totalLength * targetPercent;

  let accumulated = 0;
  const kept: string[] = [];

  for (const p of paragraphs) {
    accumulated += p.length + 2; // +2 for \n\n
    kept.push(p);
    if (accumulated >= targetLength) break;
  }

  return kept.join("\n\n");
}
```

## Open Questions

1. **audio_url Column Data Migration**
   - What we know: Currently stores full public URLs like `https://xxx.supabase.co/storage/v1/object/public/audio-files/path`
   - What's unclear: Exact format of stored URLs -- need to verify by checking actual data in the database
   - Recommendation: Write a migration that strips the URL prefix to keep only the storage path. If no audio data exists yet (dev environment), this is trivial.

2. **Trigger Role Detection Exact Mechanism**
   - What we know: `current_setting('role', true)` returns the Supabase role. Service_role key sets this differently from authenticated key.
   - What's unclear: Exact value returned for service_role -- could be `'service_role'` or it might bypass the trigger check entirely. Need to test.
   - Recommendation: Test in local Supabase instance. If `current_setting('role', true)` returns NULL for service_role, check `current_user = 'supabase_admin'` as fallback.

3. **Free Reads Counter Server-Side Tracking**
   - What we know: Currently counted client-side from `user_progress` rows. Decision says increase to 10.
   - What's unclear: Should we add a dedicated `free_reads_used` counter column on `profiles` for faster server-side checks, or just COUNT(*) from `user_progress` in the Edge Function?
   - Recommendation: COUNT(*) from `user_progress` is sufficient for the current scale. Adding a dedicated column introduces sync complexity. The Edge Function query is fast with the existing index `idx_user_progress_user_book`.
